# Deploy Targets

ForgeLM doesn't ship its own inference server — that's outside scope. Instead, `forgelm deploy` generates ready-to-run configurations for the major existing servers: Ollama, vLLM, TGI, and HuggingFace Endpoints.

## Quick example

```shell
$ forgelm deploy ./checkpoints/customer-support \
    --target ollama \
    --output ./Modelfile
✓ wrote ./Modelfile

$ ollama create my-bot -f ./Modelfile
$ ollama run my-bot
```

## Supported targets

| Target | Output | Use case |
|---|---|---|
| `ollama` | `Modelfile` | Local CPU/GPU inference, fast prototyping. |
| `vllm` | `vllm-config.yaml` | High-throughput GPU serving. |
| `tgi` | `tgi-launcher.sh` + Dockerfile | HuggingFace's text-generation-inference. |
| `hf-endpoints` | `endpoints-config.json` | One-click deploy to HuggingFace Inference Endpoints. |

KServe and NVIDIA Triton are **not** built-in targets in v0.5.5. The `forgelm deploy --target` parser accepts only the four runtimes above. Operators serving on KServe / Triton author the `InferenceService` manifest or `model_repository/` layout by hand from the GGUF / safetensors artefact.

## Ollama

```shell
$ forgelm deploy ./checkpoints/run --target ollama --output Modelfile
```

Generates:

```text
FROM ./model.q4_k_m.gguf

SYSTEM "You are a polite customer-support agent for telecom."

PARAMETER temperature 0.7
PARAMETER top_p 0.9
PARAMETER stop "<|im_end|>"

# Generated by ForgeLM 0.5.5 from checkpoint customer-support v1.2.0
```

ForgeLM picks the right `PARAMETER` lines based on the model's tokeniser (chat templates, stop tokens) — no manual configuration needed.

## vLLM

```shell
$ forgelm deploy ./checkpoints/run --target vllm --output vllm.yaml
```

Generates:

```yaml
# vllm.yaml — generated by ForgeLM
model: "/data/model"                   # mount checkpoint at /data/model
tokenizer_mode: "auto"
dtype: "bfloat16"
max_model_len: 8192
gpu_memory_utilization: 0.85
trust_remote_code: false
served_model_name: "customer-support-v1.2"

# Chat template auto-detected from tokeniser
chat_template: null                    # uses tokenizer's default
```

Run:

```shell
$ python -m vllm.entrypoints.openai.api_server --config vllm.yaml --port 8000
```

For high-throughput, ForgeLM also generates the recommended `--tensor-parallel-size` based on the model size and the configured GPU count.

## TGI (text-generation-inference)

```shell
$ forgelm deploy ./checkpoints/run --target tgi --output tgi/
$ ls tgi/
launcher.sh    Dockerfile    config.json
```

`launcher.sh`:

```shell
#!/bin/sh
text-generation-launcher \
  --model-id /data/model \
  --port 8080 \
  --max-input-length 4096 \
  --max-total-tokens 8192 \
  --dtype bfloat16 \
  --quantize bitsandbytes-nf4
```

The `Dockerfile` is multi-stage and pulls TGI's base image — push to your registry and `docker run` it.

## HuggingFace Endpoints

```shell
$ forgelm deploy ./checkpoints/run --target hf-endpoints --output endpoints.json
```

The output JSON is the body for HuggingFace's Inference Endpoints create-endpoint API:

```shell
$ curl -X POST https://api.endpoints.huggingface.cloud/v2/endpoint \
    -H "Authorization: Bearer $HF_TOKEN" \
    -H "Content-Type: application/json" \
    -d @endpoints.json
```

Or paste into HuggingFace's UI for one-click deployment.

## Configuration

```yaml
deployment:
  target: "vllm"
  served_model_name: "customer-support-v1.2"
  max_input_length: 4096
  max_total_tokens: 8192
  gpu_memory_utilization: 0.85
  chat_template: "default"              # or path to a .jinja file
  system_prompt_default: "You are..."
```

When `deployment:` is in your YAML, `forgelm` runs `deploy` automatically as the last step (after eval / safety pass).

## What's auto-detected

| Setting | Detected from |
|---|---|
| Chat template | Model's tokeniser config |
| Stop tokens | Tokeniser special tokens |
| `max_model_len` | `model.max_length` from training |
| Quantisation | Whether you ran QLoRA, exported GGUF, etc. |
| Tensor parallelism | Model size + available GPUs |

You can override any auto-detection with explicit YAML.

## Common pitfalls

:::warn
**Skipping safety eval on the deployed quant.** A q4_k_m GGUF can score worse on Llama Guard than the full-precision adapter. Re-run safety eval on the deployed artefact, not just the training output.
:::

:::warn
**Generic chat template fallback.** If your fine-tune used a custom chat template, the auto-detection might pick the base model's template (different format). Test interactively with `forgelm chat` first.
:::

:::warn
**Stale templates.** When you bump models or tokenisers, regenerate deploy configs. An old TGI config pointing at a new tokeniser silently produces malformed outputs.
:::

:::tip
ForgeLM's generated configs include a comment with the source run-id and timestamp. Always commit the generated config alongside your model release so traces back are unambiguous.
:::
